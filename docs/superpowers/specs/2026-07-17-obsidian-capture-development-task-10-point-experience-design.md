# Obsidian Capture 开发任务 10 分体验设计

状态：设计已确认，待实施
日期：2026-07-17
范围：Obsidian Capture 的“开发任务”身份、生命周期、内容结构、浏览入口、质量门和历史迁移
明确不在本规格范围：项目日报的内容与生成规则

## 1. 目标

开发任务必须同时满足两个阅读层级：

1. 打开“开发任务中心”后，用户在 10 秒内知道当天正在做哪些事情，以及每项任务的当前状态。
2. 打开单张任务卡后，首屏无需滚动即可理解任务目标、当前进展或结果、验证、阻塞和下一步。

系统还必须保持自动运行、低成本和可追溯：

- 日常 Hook、任务判断、质量检查、Markdown 渲染和 Base 聚合全部使用本地确定性能力。
- 日常产品运行时模型输入和输出 Token 均为 0。
- 只有任务终结且确定性摘要确实无法达到可读标准时，才允许进行一次可选语义整理。
- 任何模型输出都只是候选表达，不能决定任务状态、证据质量或发布资格。

## 2. 第一性原理

开发任务不是聊天记录，也不是某一天的日志。它是一个围绕稳定目标持续存在的事实主体。

一张合格的任务卡必须做到：

- **目标稳定**：任务目标不因确认、继续、重启或跨日而改变。
- **状态可信**：面向人状态、正文、验证、完成时间和发布状态一致。
- **当前优先**：首屏只显示当前有效信息，不平铺已失效结论。
- **过程可追溯**：历史变化存在于折叠审计层，并可回到本地来源。
- **人工安全**：系统重渲染不能覆盖用户补充。
- **阅读高密度**：空章节、占位语、内部 ID 和机器路径不占据日常首屏。

## 3. 任务身份与划分

### 3.1 新任务条件

只有新的独立目标才能创建开发任务。任务标题和目标分别回答：

- 任务标题：这件事叫什么。
- 任务目标：为什么做，以及做到什么程度算完成。

标题不得直接复制“开始执行”“继续”“确认”“授权确认”等聊天语句。标题采用“对象 + 动作或问题 + 阶段”的语义形式。

### 3.2 当前任务更新条件

以下输入只更新当前任务，不创建新任务：

- 确认、开始、继续、执行；
- 授权或补充约束；
- 查询当前状态；
- 要求继续测试、修复或验收同一目标；
- Codex 重启、账号切换、模型切换或上下文压缩后的同目标恢复。

同一 Codex 对话可以包含多个开发任务。出现新的独立目标时，系统必须切换任务归属，而不是把新目标追加到旧任务。

### 3.3 跨日规则

日期变化不能结束任务。

- 第二天继续同一目标时，保留原 `task_id`。
- 任务增加“跨日”显示标记和新的历史日期节点。
- 多日无活动只能进入“暂停”，不能自动完成。
- 恢复相同目标时回到原任务。
- 只有目标实质变化才创建新任务，并建立前序和后继关系。

## 4. 面向人状态体系

Obsidian 首屏和开发任务中心只显示以下状态：

| 状态 | 含义 |
|---|---|
| 进行中 | 正在执行，存在明确下一步 |
| 阻塞 | 无法继续，存在明确阻塞原因和解除条件 |
| 待验收 | 实施已完成，但仍需工具、人工、设备或业务验收 |
| 已完成 | 验收条件已满足，没有未解决阻塞 |
| 已交接 | 当前执行责任已转移，并存在明确接续入口 |
| 已取消 | 目标不再继续，并记录取消原因 |
| 暂停 | 暂时不活跃，但目标仍有效且可恢复 |

“跨日”是附加标记，不是生命周期状态。`checkpoint`、`terminal_candidate` 等内部状态不得显示在日常阅读首屏。

状态流转：

```text
进行中
├─→ 阻塞 ─→ 进行中
├─→ 暂停 ─→ 进行中
├─→ 待验收 ─→ 已完成
├─→ 已交接
└─→ 已取消
```

### 4.1 完成判定

- **可自动验收**：测试、构建、迁移或运行契约全部通过，且没有剩余阻塞和下一步，可自动进入“已完成”。
- **需要人工验收**：视觉效果、内容质量、业务认可、真实设备或其他无法由工具证明的条件，进入“待验收”，待用户确认后完成。

助手声称“已完成”不能单独作为完成证据。

## 5. 单卡双层信息架构

### 5.1 首屏阅读层

首屏只回答五个问题：

1. 当前任务的目标是什么？
2. 当前是什么状态？
3. 已经做到哪里？
4. 有什么验证或阻塞？
5. 下一步做什么？

推荐结构：

```markdown
# Obsidian Capture｜提升开发任务可读性

> [!info] 进行中 · agent · 已跨日
> 最后更新：07-18 15:40

目标：让用户在 Obsidian 中用 10 秒判断当天每项开发任务的状态、成果、阻塞和下一步。

## 当前进展

- 已完成真实文档审计
- 已确认任务划分和状态体系
- 正在确定任务卡信息结构

## 验证

- 设计规则：已确认
- 代码验证：尚未开始

## 下一步

- 完成正式规格
```

状态决定首屏顺序：

- 进行中：当前进展、下一步、验证。
- 阻塞：阻塞原因、解除条件、已完成进展。
- 待验收：待确认内容、验证证据、验收后动作。
- 已完成：最终结论、验证证据、产出链接。
- 已交接：接续入口、交接点、未完成事项。
- 暂停：暂停原因、恢复条件、最后有效进展。
- 已取消：取消原因、已产生的可复用资产。

### 5.2 首屏排版规则

- 不生成空章节。
- 不显示“无”“未形成结论”“未记录验证”等占位语。
- 每个章节最多 3 至 5 条。
- 每条只表达一件事，并尽量保持单行可读。
- 首屏正文原则上不超过 500 个中文字符。
- 旧结论不保留在首屏。
- 文件名使用“日期 - 项目 - 语义化任务名”。
- 正文 H1 不重复日期和项目。
- “我的备注”只有存在用户内容时才显示。

### 5.3 折叠审计层

首屏之后提供默认折叠的“任务历史与审计”：

```markdown
> [!abstract]- 任务历史与审计
>
> ### 07-17
> - 创建任务：提升开发任务可读性
> - 完成真实 Vault 审计
> - 确认任务划分规则
>
> ### 07-18
> - 继续同一任务
> - 状态：进行中 → 待验收
>
> ### 关键决策
> - 采用单卡双层结构
>
> ### 技术溯源
> - 任务 ID：……
> - Ledger：……
```

审计层规则：

- 只记录目标、状态、成果、验证、阻塞、决策和交接等实质变化。
- 确认、继续、普通工具调用和重复状态播报不进入时间线。
- 同一天的小更新合并为一个日期节点。
- 旧结论移入发生日期对应的历史节点。
- 没有历史、决策或任务关系时，不生成对应小节。
- 技术溯源位于最底部，默认折叠，并使用可点击链接。
- 来源冲突不删除；首屏显示冲突提示，审计层列出冲突来源和当前采用结论。

## 6. Obsidian 属性与显示

任务卡只保留约九个面向人的查询属性：

```yaml
项目: agent
状态: 进行中
开始日期: 2026-07-17
最后更新: 2026-07-18 15:40
今日活跃: 2026-07-18
跨日: true
当前进展: 正在确定任务卡设计
下一步: 完成实施规格
验证: 设计规则已确认
cssclasses:
  - obsidian-capture-task
```

任务 ID、continuity key、Ledger 路径、内部质量分和渲染版本保存在本地状态文件与折叠审计层，不占据笔记属性首屏。

增加一个仅作用于开发任务卡的本地 CSS class：

- 阅读视图默认隐藏任务卡顶部属性表。
- 状态通过正文顶部的 Obsidian callout 显示。
- Base 仍可读取属性。
- 编辑或源码模式仍可查看属性。
- CSS 关闭后 Markdown 仍完整可读。
- 不影响 Vault 中其他类型的笔记。

## 7. 开发任务中心

开发任务中心是所有开发任务的唯一日常浏览入口，不是项目日报。

默认视图“今日开发任务”显示当天发生过实质活动的任务，包括当天完成、阻塞、交接或继续开发的任务。

默认列：

| 状态 | 任务 | 项目 | 当前进展或阻塞 | 下一步 | 最后更新 |
|---|---|---|---|---|---|

排序优先级：

```text
阻塞 → 待验收 → 进行中 → 暂停 → 已完成 → 已交接 → 已取消
```

同状态按最后更新时间倒序。

附加视图：

- 进行中
- 阻塞
- 待验收
- 跨日任务
- 已完成
- 全部任务

规则：

- 跨日且今天继续开发的任务显示在“今日开发任务”。
- 昨天仍在进行但今天尚未发生实质活动的任务只显示在“进行中”。
- Base 不显示内部 ID、Ledger、路径、质量分和渲染版本。
- 点击标题进入单卡双层结构。
- “待补证”是质量提示，不新增生命周期状态。
- 空的“未命名 Base”归档；日常入口只保留一个明确命名的开发任务中心。

## 8. 内容来源与系统所有权

首屏内容只来自结构化事实：

- 目标：任务创建时的独立目标。
- 当前进展：已确认完成的实质工作。
- 当前结论：最新且仍有效的结果。
- 验证：测试、构建、运行、迁移或人工验收状态。
- 阻塞：明确的外部依赖、失败或缺失条件。
- 下一步：尚未完成且与目标直接相关的动作。
- 决策：用户明确确认，或已经落实到实现中的方案选择。

禁止保存：

- 原始聊天全文；
- 命令和补丁；
- 完整工具输出；
- 凭据、令牌和环境变量内容；
- 与任务目标无关的个人数据。

所有权：

- 系统拥有标题、状态、当前进展、结果、验证和审计时间线。
- 用户拥有“我的备注”、个人判断和补充链接。
- 系统不得修改用户区域的任何字节。
- 用户修改系统区域时，重渲染必须先检测冲突、建立备份并拒绝静默覆盖。

## 9. 本地自动质量门

质量门是本地确定性能力，不依赖模型。

检查维度：

### 9.1 完整性

- 是否存在稳定目标和面向人状态。
- 未终结任务是否存在下一步或阻塞原因。
- 已完成任务是否存在结果和验收证据。

### 9.2 可读性

- 首屏和单条内容是否超长。
- 是否存在重复、空章节、占位语或低质量标题。
- 当前层是否混入失效历史。

### 9.3 可信度

- 每条事实是否存在结构化来源。
- 模型候选摘要是否只引用允许的来源条目。

### 9.4 一致性

- 任务状态、完成时间、验证和发布状态是否匹配。
- 同一任务是否同时出现互斥状态。
- 来源摘要与渲染输入是否一致。

失败处理：

- 信息不足：显示“待补充证据”质量提示，不生成空章节。
- 新旧结论冲突：显示冲突提示并禁止正式发布。
- 人工区域异常：停止整卡重写，保留原文件并记录安全错误。
- 并发期间来源变化：放弃写入，由后续 Hook 重试。
- 语义整理失败：保留确定性版本，不静默切换模型。
- 无注册语义路由：继续零模型运行，不影响归档。

模型不能给自己打分，也不能决定任务是否完成、证据是否充分或是否允许发布。

## 10. 终结语义整理经济性规则

日期变化、静默、暂停、重启、账号切换和重复 Hook 都不能触发语义整理。

只有同时满足以下条件才允许调用一次：

1. 任务已经进入已完成、已交接或已取消。
2. 确定性首屏存在过长、冲突、多个阶段或低质量表达。
3. 存在足够的结构化来源。
4. 相同来源摘要尚未整理过。

执行顺序：

```text
结构化事实
→ 确定性首屏
→ 必要时生成候选语义摘要
→ 本地验证长度、字段、引用和禁用内容
→ 通过后展示，否则保留确定性版本
```

开发任务中心只能聚合任务卡现有字段，不得再次调用模型。

## 11. 历史迁移

历史迁移必须可逆并失败关闭：

1. 扫描任务卡、状态和来源，生成不可变来源摘要。
2. 将“确认、开始、授权、状态追问”等明显碎片并入对应真实任务。
3. 无法确定归属的内容不猜测，移入只读旧档区。
4. 为真实目标生成语义化标题和单卡双层结构。
5. 保留 Ledger、正式资产和用户备注。
6. dry-run 列出合并、保留、归档和冲突项。
7. apply 前重新验证来源摘要。
8. 所有写入建立备份。
9. rollback 恢复任务卡、Base、CSS 和映射。
10. “未命名 Base”只归档，不永久删除。

迁移不得改写既有正式 Agent10 资产正文。

## 12. 组件边界与数据流

```text
Hook 事件
→ 本地去敏与事件分类
→ 任务身份判断
→ 更新唯一任务状态
→ 本地质量门
→ 生成单卡双层 Markdown
→ 更新开发任务中心
→ 终结且确有需要时执行一次可选语义整理
```

组件职责：

- 事件层：过滤和保存允许的结构化事件。
- 任务身份层：判断新目标或当前任务继续。
- 状态层：保存当前事实、实质变化和任务关系。
- 渲染层：生成 Markdown，不判断任务是否完成。
- 质量门：执行本地验证，不改写事实。
- 语义整理层：生成候选表达，不改变状态和证据。
- Base 层：只读任务卡属性，不参与业务判断。

## 13. 验收标准

实施完成后必须证明：

1. 打开开发任务中心，用户在 10 秒内能说明今天做了什么以及每项任务状态。
2. 任一任务首屏无需滚动即可看到目标、状态、当前进展和下一步或结果。
3. 阻塞和待验收任务位于最需要关注的位置。
4. 不存在空章节、占位语和空白人工区域。
5. “确认、开始、继续、授权”不再单独创建任务。
6. 跨日、重启、账号切换和上下文压缩后仍恢复同一目标任务。
7. 当前结论与历史过程不混排。
8. 用户备注经过重复渲染后逐字节不变。
9. Base 不显示内部 ID、机器路径和渲染字段。
10. 日常运行模型 Token 为 0。
11. 终结语义整理最多一次，并存在来源摘要和执行记录。
12. 临时 Vault、真实 Vault、Hook、Agent10 和受影响消费者回归全部通过。
13. 迁移具备 dry-run、摘要校验、备份、幂等 apply 和 rollback。

## 14. 明确排除项

- 本规格不定义项目日报正文、汇总指标或生成频率。
- 本规格不选择或注册新的外部模型、提供商或数据外发范围。
- 本规格不改变 Agent10 共享监督器所有权。
- 本规格不修改既有正式资产正文。
- 本规格不引入第二份日常阅读任务卡。
